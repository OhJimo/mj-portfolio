## 1. 코드 위계질서 (현재 구조)

```
src/
├─ main.tsx                # 진입점 (StrictMode + createRoot)
├─ App.tsx                 # 셸 — Lenis + Header + main + Footer
├─ index.css               # 디자인 토큰 + base + components 레이어
├─ lib/
│  └─ utils.ts             # cn() (clsx + tailwind-merge)
├─ components/
│  ├─ header.tsx           # 스티키 내비, Lenis 스크롤 훅
│  ├─ footer.tsx           # 큰 워드마크 + 메뉴 + 이메일
│  ├─ page-guides.tsx      # 좌우 가이드 라인 (장식)
│  └─ ui/button.tsx        # shadcn Button (현재 미사용)
├─ sections/               # 섹션은 모두 named export
│  ├─ intro.tsx            # 좌/포트레이트/우 3분할 카피
│  ├─ about.tsx            # 자기소개 + 포트레이트
│  ├─ experience.tsx       # 4장의 sticky 카드
│  ├─ practice.tsx         # 캐러셀 + 본문
│  ├─ archive.tsx          # 타임라인 + 프로젝트 두 그룹
│  ├─ direction.tsx        # 좌측 헤딩 sticky + 3 블록
│  └─ closing.tsx          # 인사말 + 이메일 CTA
└─ assets/
   ├─ practice/            # 1개 (5.7MB)
   └─ projects/            # 6개 (총 9.7MB)
```

**위계 평가**
- 진입 → App → (Header / Sections / Footer) 의 1-depth 단순 구조. **KISS 만족.**
- 데이터(콘텐츠 배열)와 뷰가 같은 파일에 있는 섹션이 다수 (`experience`, `archive`, `direction`, `header`, `footer`). 콘텐츠 분량이 작고 한 번만 쓰이므로 **현 시점에서는 분리 불필요**(KISS).
- `components/ui/`는 shadcn 관습에 맞춰 분리되어 있으나 실사용처 없음.
- `lib/`는 `cn`만 들어 있는 상태. 추가 유틸이 생기면 같은 자리에서 확장 가능.

## 3. 디자인 시스템 — 토큰의 "정의-소비" 단절 (가장 큰 디자인 부채)

> `index.css`에는 타이포·간격·라디우스·모션·컬러 토큰이 의도적으로 짜여 있다. 그러나 7개 섹션 중 6개가 토큰을 우회하고 인라인 `clamp()`·임의 라디우스·임의 투명도로 직접 그려진다. 시스템이 "참고용 메모"로 전락했다.

### 3.1 타이포 토큰 — 정의됨, 사용 0회

`index.css:119-124`
```css
--type-display: 3.5rem;
--type-h1:      2.5rem;
--type-h2:      1.875rem;
--type-h3:      1.375rem;
--type-body:    1.0625rem;
--type-small:   0.875rem;
```
→ 컴포넌트에서 사용된 횟수: **0회**. 대신 인라인 `clamp()` 16곳에서 9가지 변형이 떠돈다.

| 등장 표현식 | 등장 위치 | 분류 추정 |
|---|---|---|
| `clamp(2.5rem, 5vw, 4rem)` | about · archive · direction · experience · practice (5곳) | 섹션 타이틀 |
| `clamp(4.5rem, 12vw, 8rem)` | experience (배경 숫자) | 디스플레이 |
| `clamp(1.5rem, 4.5vw, 3.75rem)` | intro ×2 | 디스플레이 |
| `clamp(1.5rem, 3vw, 2.25rem)` | experience | h3 |
| `clamp(1.5rem, 3vw, 2rem)` | archive | h3 |
| `clamp(1.5rem, 2.5vw, 2rem)` | about | h3 |
| `clamp(1.3rem, 2.1vw, 1.75rem)` | closing | 강조 본문 |
| `clamp(1.5rem, 2.6vw, 2rem)` | footer | 강조 본문 |
| `clamp(0.95rem, 1.1vw, 1.25rem)` | intro 서브카피 | small-lg |

**문제**: 같은 "h3"가 세 가지 정의(`2.25rem` / `2rem` / `2rem`)로 미묘하게 어긋남. "강조 본문"도 `1.75rem`과 `2rem`. 디자이너가 "헤딩 한 단계 키우자"라고 결정해도 단일 진입점이 없음.

**제안**
1. **의도(intent) 기반 토큰** 추가: `--text-section-title`, `--text-display-xl`(intro), `--text-card-title`, `--text-eyebrow`.
2. Tailwind v4의 `@theme inline`에 등록 → `text-section-title` 유틸로 소비.
3. clamp의 max를 통일된 스케일에 맞춤 (예: 4.0 / 3.0 / 2.25 / 1.75 / 1.25rem).

### 3.2 라디우스 토큰 — 스케일이 실제 디자인의 1/2

`index.css:66-72`
```css
--radius:     0.45rem;
--radius-sm:  ≈ 0.27rem
--radius-md:  ≈ 0.36rem
--radius-lg:  = 0.45rem
--radius-xl:  ≈ 0.63rem
--radius-2xl: ≈ 0.81rem
--radius-3xl: ≈ 0.99rem
--radius-4xl: ≈ 1.17rem
```

→ 컴포넌트의 실제 사용:
- 카드 컨테이너: `rounded-2xl`(1rem · Tailwind 기본)
- 큰 카드: `rounded-[2rem]`, `rounded-[1.95rem]`, `rounded-[1.75rem]` — 임의 값 3종
- 버튼: `rounded-md`, 도트: `rounded-full`

**문제**: 정의된 토큰은 0.27~1.17rem 범위, 실제 사용은 1.0~2.0rem 범위. **토큰 스케일이 실제 디자인의 1/2 수준.** 같은 "큰 카드"가 세 가지 라디우스로 갈림.

**제안**: 의도 기반 추가 — `--radius-card: 1rem`, `--radius-frame: 1.75rem`, `--radius-frame-lg: 2rem`. 캐러셀 내부 `1.95rem`은 외곽 `2rem` 안의 시각 정렬용 — 주석으로 의도 명시.

### 3.3 간격 토큰 — 죽은 토큰

`index.css:136-141`에 `--stack-xs/sm/md/lg/xl/2xl` 정의. 컴포넌트는 모두 Tailwind `space-y-*` / `gap-*`로 직접 표기. **토큰 사용 0회.**

**판단**: Tailwind v4가 spacing 자체를 토큰으로 다루므로 큰 문제는 아님. 다만 `--stack-*`은 **죽은 토큰** → 삭제 또는 활용 결정 필요.

### 3.4 컬러·투명도 어휘 — 무질서

| 색·투명도 | 등장 횟수 | 위치 | 의도 |
|---|---:|---|---|
| `white/15` | 1 | footer 디바이더 | 분리선 |
| `white/35` | 2 | practice 화살표 보더 | 컨트롤 윤곽 |
| `white/55` | 4 | footer 라벨, practice 도트 | 보조 텍스트 / 비활성 도트 |
| `white/65` | 1 | footer 작가명 보조 | 부수 정보 |
| `white/70` | 2 | footer 호버 | 호버 상태 |
| `white/85` | 1 | footer copyright | 일반 본문 |
| `black/5` | 2 | experience/archive 카드 보더 | 카드 윤곽 |
| `black/6` | 2 | practice/closing 프레임 보더 | 카드 윤곽 |
| `black/18` | 1 | closing 그라데이션 | 페이드 약 |
| `black/22` | 1 | practice 그라데이션 | 페이드 강 |
| `foreground/90` | 1 | practice 본문 강조 | — |
| `foreground/95` | 1 | closing 본문 강조 | — |

**문제**
- "카드 보더"가 `black/5`와 `black/6` — 1% 차이, 의도 불명.
- "그라데이션 페이드"가 `/18`과 `/22` — 시각적으로 거의 동일.
- footer 안에서 white 투명도 6단계(15/35/55/65/70/85). 위계가 드러나지 않음.

**제안 — 6단계 의미 기반 스케일**
```css
--alpha-line:         0.06   (보더, 디바이더)
--alpha-overlay-soft: 0.18   (그라데이션 약)
--alpha-overlay:      0.35   (컨트롤 윤곽, 그라데이션 강)
--alpha-muted:        0.55   (보조 텍스트, 비활성)
--alpha-secondary:    0.75   (호버, 부속 정보)
--alpha-primary:      0.90   (본문)
```

### 3.5 올리브 색 시스템 — 시그니처 컬러가 토큰에 없다

정의된 토큰: `--olive-line`, `--olive-line-soft`, `--olive-accent`.
누락: 올리브 서피스 `oklch(0.4 0.09 107.4)` ← **footer 배경 + experience olive 카드** (둘 다 인라인 `style`).

→ "두 번의 의도된 강세"의 색이 시그니처인데 토큰화돼 있지 않다. **반드시 같은 변수 하나로 묶어야 한다** (footer와 experience가 늘 같이 변하도록).

권장 5단계 정리:
```css
--olive-50:  oklch(0.97 0.025 107.4)   (얼라이트 톤, 카드 배경 light)
--olive-100: oklch(0.94 0.035 107.4)   (= --olive-line-soft)
--olive-300: oklch(0.88 0.06  107.4)   (= --olive-line)
--olive-500: oklch(0.50 0.12  107.4)   (= --olive-accent, 링크)
--olive-700: oklch(0.40 0.09  107.4)   (서피스 — footer & experience olive 카드 공유)
```

### 3.6 본문 대비 (WCAG)
- `muted-foreground` = `oklch(0.58 0.031 107.3)` 흰 배경 위 → 대비 ≈ **4.6:1**, AA 턱걸이/AAA 미달. 보조 텍스트(intro 서브카피, archive 연도, footer 라벨)에 부담. → **0.58 → 0.50**으로 진하게 권장.
- footer의 `text-white/55` 올리브 위 → 대비 ≈ **2.7:1**, **AA 미달**. footer 라벨(Email/Menu/Copyright)은 최소 `white/70` 이상으로.

## 4. 타이포그래피 — weight 모순과 시맨틱

### 4.1 weight 모순 (S-Core Dream에 700이 없음)
로드된 S-Core Dream weight = **400 / 500 / 600**. 그런데 섹션 타이틀은 `font-bold`(700)을 씀.
- 브라우저가 가장 가까운 600을 쓰거나
- **합성 굵게(faux bold)** 처리 — 글자 형태가 뭉개짐.

`index.css:240-244`의 전역 `h1~h6`은 `font-weight: 500`으로 잘 정의돼 있으나, 섹션 컴포넌트가 매번 `font-bold`로 덮어쓴다.

**선택지**
- (A) S-Core Dream **Bold(700)** 추가 로드 — 다운로드 비용 ↑.
- (B) 섹션 타이틀을 `font-semibold`(600)로 통일 + 토큰화.
- (C) 헤딩 디폴트(500) 유지 — 가장 차분한 에디토리얼 톤.

→ **(C) 또는 (B) 권장.** 본문이 차분해야 "두 번의 의도된 강세"(10장)가 살아난다.

### 4.2 의미 vs 표현
`about.tsx:6` "소개" = `<p>`. 다른 섹션 = `<h2>`. 같은 디자인 의도인데 시맨틱 어긋남. SR/SEO 손해. **`<h2>`로 통일** (스타일은 동일 유지).

### 4.3 본문 측정값 (잘 짜여 있음)
- `--type-body: 17px`, `--leading-body: 1.75`, `--reading-max: 40rem`
- 한국어 본문에 매우 적절. **이 부분은 그대로 유지.**

### 4.4 마루부리 — 제거됨 ✅
2026-04-28 결정·실행 완료. `--font-emphasis` / `.font-emphasis` / `@import "@kfonts/maruburi/..."` / `package.json` 의존성 모두 삭제. 빌드 산출물에서 woff/woff2 5종 약 9MB 사라짐. 향후 강조는 **굵기·자간·올리브 강조색·라인** 등 기존 시스템 안에서 해결.

## 5. 첫 인상 — Header + Intro의 placeholder 동시 노출

## 9. 섹션별 디자인 노트

### Intro (2026-04-28 갱신)
*매직 픽셀 오프셋은 이미 해결됨* — 좌우 컬럼이 `minHeight: CARD_HEIGHT` + `md:absolute md:top-1/2 md:-translate-y-1/2`로 정렬되고, 헤드와 서브카피가 한 래퍼 안에 묶임. 남은 진짜 문제는 다음:

- **좌우 결속 기법 비대칭** (`intro.tsx:14-24` vs `40-50`)
  - 좌측: 서브카피가 `md:absolute md:bottom-full` + `marginBottom: SIDE_SUB_COPY_GAP` (헤드 위로 띄움)
  - 우측: 서브카피가 자연 흐름 + `mt-3` + `marginTop: SIDE_SUB_COPY_GAP` (헤드 아래로 깔림, `mt-3`과 인라인 marginTop이 겹침)

## 11. Practice 캐러셀 — 디자인 + 기능 종합

### 11.1 🔴 autoplay 의도 vs 실제 동작 (기능 결함 후보)
`src/sections/practice.tsx:42-55`
```tsx
useEffect(() => {
  if (isPaused) return
  const intervalId = window.setInterval(() => { ... }, AUTOPLAY_INTERVAL_MS)
  return () => window.clearInterval(intervalId)
}, [isPaused, displayIndex])  // ← displayIndex가 deps
```
`displayIndex`가 deps에 있어서 **슬라이드가 바뀔 때마다 인터벌이 cleanup → 재설치**된다.
- 사용자가 화살표/도트로 수동 조작 → 다음 자동 재생까지 5초가 새로 카운트 (의도일 수도).
- 그러나 `setDisplayIndex((prev) => prev + 1)` 함수형 업데이트로 충분 → deps는 `[isPaused]`만으로 동등 동작. **불필요한 의존성**.
- StrictMode 더블 마운트와 결합 시 한 번 더 발생.

**권장 패치**: `[isPaused]`로 단순화.

### 11.2 🟡 빠른 다중 클릭 race

### 11.3 키보드 네비게이션 부재

## 12. 모션·인터랙션의 접근성

### 12.1 `prefers-reduced-motion` 미처리
- Lenis는 옵션 제공하지만 미적용.
- Practice 캐러셀 autoplay 5초마다.
- Hover-image 스케일 1.02, hover-card translateY -2px 등.

→ vestibular 민감 사용자에게 부담. 다음 추가:
```css
@media (prefers-reduced-motion: reduce) {
  html { scroll-behavior: auto; }
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```
+ 캐러셀 autoplay `if (prefersReducedMotion) return`.
+ `<ReactLenis options={{ ... }} />` 조건부 비활성.

### 12.2 Hover에만 의존하는 affordance
- 캐러셀 화살표·카드 lift 효과 모두 호버 의존 → 모바일은 정적/흐린 상태만. 11.5 참조.

### 12.3 Lenis + native smooth scroll 중복
`index.css:221`의 `html { scroll-behavior: smooth }`와 Lenis 동시 활성. 공식 가이드는 native smooth 끄기 권장 (충돌로 미세 jank). → **`scroll-behavior: smooth` 제거.**
