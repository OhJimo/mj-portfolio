# 포트레이트 모핑 재구현 문서

## 문서 목적

메인 페이지 **섹션 1(Intro)** 의 포트레이트 이미지가 스크롤에 따라 **섹션 2(About)** 의 이미지 자리로 이동하는 인터랙션을 처음부터 다시 정의한다.

이번 문서는 다음을 명확히 고정한다.

1. 레퍼런스는 참고하되, 구현 범위는 **섹션 1 → 섹션 2** 로 압축한다.
2. 애니메이션은 **framer-motion을 유지**한 채 재구현한다.
3. 실제로 움직이는 카드는 **단일 floating 카드 1개**다.
4. 실제 이미지 상태는 **2개만** 사용한다.
   - 섹션 1 포트레이트
   - 섹션 2 포트레이트

중간 `workspace` 이미지는 이번 구현에서 사용하지 않는다.

---

## 이번 문서에서 버리는 것

이전 문서와 구현에서 혼선을 만든 요소는 이번 문서 기준으로 폐기한다.

- `X축 / Y축` 같은 표현 중심의 설명
- 360도 full spin을 기본값으로 삼는 접근
- `workspace`를 반드시 거쳐야 한다는 가정
- `framer-motion` 자체가 문제라고 보는 판단
- 정적 카드와 플로팅 카드가 함께 보이는 구조

이번 문서에서는 축 이름보다 **시각 행동**으로 설명한다.

- `좌우 회전`
- `가장 얇아지는 구간`
- `이미지 교체`
- `섹션 2 자리로 안착`

구현 레벨에서는 이 동작을 **CSS `rotateY` 계열**로 해석한다.

---

## 표준 CSS 축 용어 매핑

축 용어 혼선이 이 인터랙션 작업 중 가장 큰 시간 손실의 원인이었다. 향후 같은 혼선이 재발하지 않도록 표준 CSS 정의와 멘탈 모델을 명시적으로 매핑한다.

### 표준 CSS 정의

| 축 | 회전축 (꼬챙이) | 시각 동작 |
|---|---|---|
| `rotateX` | 가로축 (수평선) | 위/아래로 숙임 (책상 위 종이 들어올리기) |
| `rotateY` | 세로축 (수직선) | 좌/우로 회전 (회전문) |
| `rotateZ` | 화면을 뚫고 나오는 축 | 평면 시계/반시계방향 회전 |

### 멘탈 모델 → 표준 CSS

| 사용자/구현자 표현 | 실제 의도 동작 | 표준 CSS |
|---|---|---|
| "X축으로 빙글" / "X축 회전" | (대부분) 좌우 회전 | **rotateY** |
| "좌우 회전" / "회전문 같은" | 좌측 면이 가까이 ↔ 우측 면이 멀어짐 | rotateY |
| "위아래 회전" / "책상에서 종이 들기" | 윗변이 멀어짐 ↔ 아랫변이 가까워짐 | rotateX |

### 결론

이 문서와 코드는 **표준 CSS 정의를 따른다**: `rotateY = 좌/우 회전`. 사용자 의도와 표준 정의가 이 인터랙션에서는 일치한다.

향후 문서 작성 시 "X축/Y축" 명칭만으로 설명하지 말고 **반드시 시각 행동(좌우/위아래)을 함께 적는다**.

## 레퍼런스 해석

상세 첨부 이미지 세트:
`docs/섹션 1,2 이미지 애니메이션 이미지/스크린샷 2026-04-28 오전 8.59.39.png` ~ `오전 9.00.21.png`

이 이미지 흐름이 실제로 보여주는 것은 다음이다.

1. 카드는 섹션 1 중앙에서 **정면에 가깝게 시작**한다.
2. 스크롤이 진행되면 카드가 **좌우 방향으로 얇아지듯 회전**한다.
3. 카드가 가장 얇아지는 구간에서 **이미지 내용이 교체**된다.
4. 그 다음 카드가 섹션 2 우측 자리로 이동하며 **안착**한다.
5. 최종 상태는 완전한 옆면이나 뒷면이 아니라, **안정된 카드 자세**다.

중요한 점은 레퍼런스의 본질이 `한 바퀴 도는 회전`이 아니라는 점이다.

핵심은:
- 회전 그 자체
- 가장 얇아지는 순간
- 그 순간의 이미지 전환
- 우측 안착

즉, 이번 구현은 레퍼런스를 이렇게 압축해 가져온다.

### 레퍼런스 압축 규칙

- 원본 레퍼런스: 3개 섹션에 걸친 긴 전환
- 우리 구현: 2개 섹션 사이의 짧은 전환

압축 후 남는 구조:

1. 섹션 1 포트레이트
2. 좌우 회전으로 얇아지는 구간
3. 섹션 2 포트레이트로 교체
4. 섹션 2 자리로 안착

### 12 프레임 상세 분석 (원본 레퍼런스, 3섹션 기준)

레퍼런스 사이트(Joseph Robert)의 12장 스크린샷을 프레임 단위로 분석한 결과. 우리 구현은 이 중 0~50% 구간(첫 transition까지)을 우리 2섹션에 압축해서 가져온다.

| 프레임 | 진행도 | rotateY 추정 | 카드 모양 | 이미지 |
|---|---|---|---|---|
| 1 | 0% | 0° | 정면, 평평 | portrait |
| 2 | ~10% | -10°~-15° | 살짝 틸트 | portrait |
| 3 | ~30% | -25°~-30° | 강한 틸트 | portrait |
| 4 | ~45% | **-50°~-60°** | 거의 edge-on | (전환 중) |
| 5 | ~55% | -40°~-50° | 다시 열림 | **workspace 등장** |
| 6 | ~65% | -25°~-30° | 안정적 틸트 | workspace |
| 7 | ~72% | -15°~-20° | 살짝 틸트 | workspace |
| 8 | ~78% | -20°~-25° | 다시 기울기 시작 | workspace |
| 9 | ~85% | **-50°~-60°** | edge-on | (전환 중) |
| 10 | ~90% | -30°~-35° | 다시 열림 | **portrait 복귀** |
| 11 | ~95% | -15°~-20° | 약한 틸트 | portrait |
| 12 | 100% | -10°~-15° | 안착 | portrait |

핵심 패턴:
- **두 번의 깊은 틸트 피크**(45%, 85%)에서 카드가 가장 좁아지면서 이미지 swap이 가려짐
- 회전축은 명백히 rotateY (좌우 회전)
- 90°를 넘지 않으므로 backface 노출은 없음
- 위치는 처음부터 끝까지 점진적 이동

우리 구현(2섹션)에선 피크가 1회만 필요하므로 0~50% 구간 패턴 한 번만 적용 → `rotateY: [0, -18, -34, -20, -10]` 형태로 압축.

## 최종 컨셉

`섹션 1의 중앙 포트레이트가 하나의 카드처럼 움직이며, 스크롤 중간에 얇아지는 순간 섹션 2의 포트레이트로 바뀌고, 섹션 2의 이미지 자리에 도착한다.`

이 컨셉에서 가장 중요한 원칙은 두 가지다.

1. 사용자는 항상 **한 장의 카드만** 본다.
2. 이미지 전환은 **가장 얇아지는 순간**에 일어난다.

## 시각 동작 정의

### 1. 시작 상태

- 섹션 1 중앙 포트레이트가 보인다.
- 카드는 정면에 가깝다.
- 섹션 2의 정적 이미지는 자리만 유지하고 실제로는 보이지 않는다.

### 2. 회전 시작

- 스크롤이 조금 진행되면 카드가 `좌우 회전`을 시작한다.
- 이동도 처음부터 점진적으로 함께 진행되지만, 초반엔 회전 변화량이 위치 변화량보다 크기 때문에 **회전이 더 두드러져 보이는 인상**으로 읽힌다.
- 즉 "정지 중 회전"이 아니라 "이동 중 회전"이다 — 위치는 분리 없이 흐른다 (동시 점진적 원칙 일관 유지).

### 3. 얇아지는 구간

- 회전이 깊어지며 카드 폭이 시각적으로 가장 얇아진다.
- 바로 이 구간에서 섹션 1 포트레이트가 사라지고, 섹션 2 포트레이트가 나타난다.
- 이 전환은 하드컷보다 **짧은 크로스페이드**가 좋다.

### 4. 이동 구간

- 카드는 처음부터 끝까지 점진적으로 이동한다 (동시 점진적).
- 회전이 깊어지는 동안에도 위치는 일정한 속도로 섹션 2 쪽으로 흐른다.
- 카드가 너무 오랫동안 옆면 상태를 유지하지 않게 한다.

### 5. 안착 상태

- 카드가 섹션 2 자리로 도착한다.
- 최종 자세는 완전한 정면이 아니어도 되지만, 읽기 어려운 측면 상태는 피한다.
- 최종 이미지는 섹션 2 포트레이트다.

## 진행도 설계

스크롤 진행도는 `0 ~ 1`로 관리한다.

- `0`: 섹션 1 시작
- `1`: 섹션 2 이미지 자리 안착 직전

### 회전 곡선 4구간

| 구간 | 진행도 | 역할 |
|---|---|---|
| A | 0.00 ~ 0.32 | 시작 유지 + 살짝 기울기 시작 |
| B | 0.32 ~ 0.58 | 가장 얇아지는 구간으로 진입 |
| C | 0.56 ~ 0.70 | 가장 얇은 구간 + 이미지 교체 |
| D | 0.70 ~ 1.00 | 회전 완화 + 섹션 2 자리에 안착 |

### 위치 이동 — 동시 점진적

위치는 **회전과 분리하지 않고 0.00 ~ 1.00 전체 구간에서 동시 점진적으로 진행**한다. 레퍼런스도 위치는 처음부터 부드럽게 흐른다. 회전이 끝난 뒤에야 카드가 미끄러지는 분리형 타이밍은 채택하지 않는다.

### 구간별 기대 인상

- **A**: 시작 이미지를 안정적으로 읽으며 카드가 천천히 기울기 시작.
- **B**: 카드가 얇아지면서 이동도 함께 진행됨.
- **C**: 이미지가 바뀌는 핵심 구간. 카드가 가장 좁아 swap이 가려짐.
- **D**: 회전이 펴지며 섹션 2 자리로 부드럽게 안착.

## 1차 구현 권장값

이번 문서는 수치를 과하게 복잡하게 두지 않는다. 먼저 읽히는 동작을 만드는 것이 목표다.

### 회전

구현은 `rotateY`를 사용한다.

권장 1차값:

```ts
rotateY: [0, -18, -34, -20, -10]
```

매핑 예시:

```ts
progress: [0.00, 0.32, 0.58, 0.82, 1.00]
```

의도:
- 초반은 거의 평면
- 중간에서 가장 강하게 얇아짐
- 후반에 회전 완화
- 완전한 full spin은 피함

**왜 -34°인가 (레퍼런스 -50°~-60° 대비 보수적 압축값)**

레퍼런스 12 프레임 분석에선 피크가 -50°~-60°였지만, 우리 1차 구현은 의도적으로 **-34°로 보수적으로 시작**한다. 같은 값이 아닌 이유:

- **레퍼런스의 피크가 깊은 건 3섹션에 걸친 긴 전환에서 두 번의 깊은 피크를 만들기 위함**. 우리는 2섹션 1피크 구조라 동일 깊이가 필요하지 않음.
- 첫 검증에서 너무 격렬하면 시각 손실(찌그러짐 인상) 위험 → **보수적으로 시작 후 부족하면 단계적 증가**.
- swap 가려짐이 부족하다고 판단되면 -45° → -55°로 키우는 순서 권장. 처음부터 -55°로 가지 않음.

### 보조 회전

카드가 너무 기계적으로 보이지 않게 아주 약한 2D 회전만 추가한다.

```ts
rotateZ: [0, 0.4, 1.2, 2.0]
progress: [0.00, 0.58, 0.82, 1.00]
```

### 위치 이동

위치는 회전과 함께 **동시 점진적**으로 진행한다.

```ts
x: useTransform(scrollY, [0, end], [sx, ex])
y: useTransform(scrollY, [0, end], [sy, ey])
```

레퍼런스도 위치는 처음부터 끝까지 일정한 속도로 흐른다. 회전이 끝난 뒤 옆으로 미끄러지는 분리형 타이밍은 인상이 깨져서 사용하지 않는다.

## 이미지 교체 규칙

이미지는 총 2장이다.

- `intro portrait`
- `about portrait`

교체 규칙:

- `0.00 ~ 0.56`: intro 100 / about 0
- `0.56 ~ 0.70`: intro 100 → 0 / about 0 → 100
- `0.70 ~ 1.00`: intro 0 / about 100

핵심은:
- 회전이 가장 깊은 구간에서 교체가 일어나야 한다.
- 이동이 다 끝난 뒤 교체하면 늦다.
- 너무 이른 교체도 어색하다.

## 레이어 구조

### 기본 원칙

- Intro와 About에는 각각 `anchor`만 남긴다.
- 실제로 보이는 카드는 `floating 카드 1개`다.
- 정적 anchor 이미지는 morph 활성 중 시각적으로 제거한다.

### 정적 anchor 처리

정적 anchor는 아래 역할만 한다.

- 레이아웃 유지
- 시작/도착 좌표 측정 기준

권장 방식:

- wrapper는 유지
- 실제 정적 이미지는 `opacity: 0`
- wrapper 배경/회전은 morph 활성 중 최소화

중요한 것은 사용자가 **두 장의 카드를 동시에 보지 않게 하는 것**이다.

## 기술 선택

이번 구현은 `framer-motion`을 유지한다.

이유:
- 현재 문제의 본질은 라이브러리보다 설계 문제였다.
- `useScroll`, `useTransform`만으로도 이번 2-state 전환은 충분히 구현 가능하다.
- 지금 단계에서 라이브러리 교체는 리팩토링 비용이 크고, 이득이 불분명하다.

즉,
- 이번 문서는 `비프레이머`가 아니라
- `프레이머를 단순하게 쓰는 재설계 문서`다.

## 구현 구조

```tsx
<PortraitMorphProvider>
  <Header />
  <main ref={mainRef} className="relative">
    <IntroSection />
    <AboutSection />
    ...
    <MorphingPortrait />
  </main>
</PortraitMorphProvider>
```

### 역할 분리

- `IntroSection`
  - 시작 anchor 제공
  - morph 활성 중 정적 이미지는 보이지 않게 처리
- `AboutSection`
  - 도착 anchor 제공
  - morph 활성 중 정적 이미지는 보이지 않게 처리
- `MorphingPortrait`
  - 실제 카드 렌더링
  - 회전, 이동, 이미지 교체 담당

## 구현 원칙

### 1. width / height 보간 제거

인트로와 소개 섹션 이미지 슬롯 크기가 이미 비슷하므로, 1차 구현에서는 크기 보간을 제거한다.

즉:
- 카드 크기는 고정 또는 거의 고정
- 먼저 회전/교체/이동만 읽히게 만든다.

### 2. 모바일 비활성

모바일에서는 이 모핑을 끈다.

이유:
- 섹션 구조가 세로 스택으로 바뀌어 경로 인상이 약해진다.
- 작은 화면에서는 과한 움직임보다 정적 이미지가 더 낫다.

기준:
- `min-width: 768px` 이상에서만 활성

### 3. backface 처리

카드가 가장 얇아지는 구간에서 뒷면이 이상하게 비치지 않게 해야 한다.

권장:

```css
backface-visibility: hidden;
transform-style: preserve-3d;
```

다만 이번 구현은 full spin을 쓰지 않기 때문에, `backface`가 주인공이 되는 구조는 아니다.

## 단계별 구현 순서

이번 구현은 한 번에 끝내지 않는다. 기능을 얹을 때마다 바로 검토할 수 있도록, 아래 6개 phase로 나누어 진행한다.

### Phase 0 — 인프라만 연결

목표:
- `PortraitMorphProvider` 연결
- `MorphingPortrait` 연결
- `Intro / About` anchor ref 연결
- floating 카드 1장을 **섹션 1 시작 자리**에만 띄운다
- 이 단계에서는 **모션 없음**

중요:
- `about placeholder`는 **보이게 둔다**
- 이건 최종 상태가 아니라, 이후 phase에서 안착 위치를 검증하기 위한 임시 기준점이다

검증 포인트:
- 모핑 카드가 intro placeholder 자리에 정확히 겹치는가
- 화면이 깨지지 않는가
- 모바일에서는 모핑 카드가 뜨지 않는가

### Phase 1 — 위치 이동만

목표:
- `x / y`만 추가
- `scrollY [0, end] → [sx, ex] / [sy, ey]`
- 회전 없음
- 이미지 교체 없음

중요:
- 카드가 **직선 경로로** 섹션 1 → 섹션 2 자리로 이동하는지만 본다
- `about placeholder`는 계속 보이게 둔다

검증 포인트:
- 스크롤하면 카드가 intro → about 자리로 자연스럽게 이동하는가
- about placeholder 위에 정확히 안착하는가
- 100% 시점에 이동이 멈추는가

### Phase 2 — 좌우 회전만

목표:
- `rotateY`만 추가
- 문서의 1차값 사용
  - `rotateY: [0, -18, -34, -20, -10]`
  - `progress: [0.00, 0.32, 0.58, 0.82, 1.00]`
- 위치 이동은 유지
- 이미지 교체 없음

중요:
- 이 단계의 핵심은 `좌우 회전으로 읽히는가`다
- 아직 swap은 보지 않는다

검증 포인트:
- 회전이 위아래가 아니라 **좌우 회전**으로 읽히는가
- 피크 시점에 카드가 가장 얇아지는가
- 마지막 자세가 약 `-10°` 근처로 안착하는가
- 찌그러짐이나 이상한 full spin 인상이 없는가

### Phase 3 — 이미지 교체만

목표:
- `introOpacity`, `aboutOpacity` 추가
- `0.56 ~ 0.70` 구간에서 cross-fade
- 카드가 가장 얇아지는 순간 이미지가 바뀌게 한다

중요:
- `workspace`는 쓰지 않는다
- `intro → about` 2-state 교체만 본다

검증 포인트:
- swap이 얇아지는 순간 자연스럽게 가려지는가
- swap 시점이 너무 빠르거나 늦지 않은가
- 바뀐 이미지가 정확히 섹션 2 포트레이트인가

### Phase 4 — 정적 placeholder 처리

목표:
- `hideStaticImage` 로직 활성화
- intro / about 정적 placeholder를 morph 활성 중 숨긴다
- 최종적으로 사용자는 **floating 카드 한 장만** 보게 한다

중요:
- 앞선 phase에서는 검증 편의를 위해 placeholder를 남겨두지만,
  **최종 동작에서는 숨김**이 맞다
- 즉, 이 문서의 최종 spec과 단계별 검증 방식은 구분한다

검증 포인트:
- 데스크톱에서는 floating 카드 한 장만 보이는가
- 모바일에서는 두 정적 이미지가 각자 자리에 정상적으로 보이는가
- 리사이즈 시 상태 전환이 자연스러운가

### Phase 5 — 폴리시

목표:
- `rotateZ: [0, 0.4, 1.2, 2.0]` 추가
- `borderRadius: [16, 32]` 추가
- easing / 미세 수치 튜닝

중요:
- 본질 동작이 맞은 뒤에만 진행한다
- 이 단계는 없어도 인터랙션 자체는 성립한다

검증 포인트:
- 카드가 너무 기계적으로 보이지 않는가
- 모서리 변화가 자연스러운가
- 최종 인상이 레퍼런스와 충돌하지 않는가

### 왜 이 순서인가

1. **인프라 먼저**
   - 카드가 제대로 뜨지 않으면 모션을 검증할 수 없다
2. **위치 먼저**
   - 안착 좌표가 틀리면 회전 디버깅이 불가능하다
3. **회전 다음**
   - swap은 회전 피크 지점에 의존하므로, 회전 곡선이 먼저 정리돼야 한다
4. **swap 그 다음**
   - swap이 자연스러운지 보려면 회전이 이미 읽혀야 한다
5. **placeholder 숨김은 마지막 본질 단계**
   - 초반엔 검증 보조 도구로 쓰고, 최종 단계에서만 숨긴다
6. **폴리시는 가장 마지막**
   - `rotateZ`, `borderRadius`는 본질이 아니라 디테일이다

## Lesson Learned

이전 4번의 시도에서 얻은 교훈. 향후 유사한 인터랙션 작업 시 같은 함정에 다시 빠지지 않도록 정리.

### 1. 축 용어 함정

문제: 사용자가 말한 "X축 회전"을 표준 CSS의 `rotateX`로 그대로 해석 → 위/아래 회전이 구현됨. 사용자 의도는 좌우 회전(`rotateY`)이었음.

교훈: 인터랙션 작업에서 **축 명칭은 신뢰하지 말 것**. 반드시 시각 행동으로 재확인 ("좌우로 도는 거예요? 위아래로 도는 거예요?"). 멘탈 모델과 표준 CSS 정의는 자주 어긋난다.

### 2. Full Spin (360°) 함정

문제: "빙글 도는" 표현을 360° 한 바퀴로 해석 → mid-rotation에서 perspective 압축으로 카드가 찌그러짐. backface 처리가 없으면 거꾸로 뒤집힌 이미지까지 노출됨.

교훈: 자연스러운 회전 인상은 **180° 미만의 비단조 틸트**가 가장 안전. Full spin은 카드가 edge-on을 거치며 시각 손실이 큼. 사용자가 "빙글"이라고 표현해도 실제 원하는 건 "회전 인상이 분명한 틸트"인 경우가 많다.

### 3. framer-motion ≠ 문제 원인

문제: 인터랙션이 의도대로 안 나오자 라이브러리 책임론 등장 → "비프레이머로 다시 짜자".

교훈: 라이브러리는 도구일 뿐. 실제 문제는 **회전 곡선 / backface 처리 / 이동 타이밍 / 축 해석** 같은 설계 요소. 라이브러리 교체 전에 설계 요소부터 점검할 것. 라이브러리 교체는 큰 비용에 비해 동일 문제를 다른 모양으로 재발시킬 가능성이 높다.

### 4. 정적 placeholder 표시는 디버깅 임시 조치

문제: 모핑 카드가 엉뚱한 자리에 안착 → 사용자가 "placeholder를 표시해서 안착 위치를 시각 검증하고 싶다" 요청 → 운영 단계에서도 표시한 채로 두려는 유혹.

교훈: 개발/디버깅 단계의 시각적 보조 장치는 **운영 spec에 섞이면 안 됨**. 디버깅이 끝나면 원래 원칙(단일 floating 카드)으로 복귀할 것.

### 5. 의사결정 동결의 어려움

문제: 결정이 바뀔 때마다 새 문서 작성 → 4개 문서가 서로 모순됨. 어느 문서가 truth인지 불명확해지면서 후속 구현이 매번 잘못된 문서를 참조.

교훈: **단일 SSOT 문서를 유지**하고 그 안에서 결정 변경 history를 관리할 것. 새 문서 신설보다 기존 문서 갱신이 안전. SSOT가 흔들리면 모든 후속 작업이 흔들린다.
