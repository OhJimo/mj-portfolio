# 포트레이트 모핑 구현 작업 문서

## 문서 목적

이 문서는 `포트레이트 모핑` 인터랙션의 **구현 자체**를 완성하기 위한 작업 문서다.

가장 중요한 원칙은 다음과 같다.

1. **도달지는 문서가 아니라 구현 결과물이다.**
2. 문서는 구현을 돕는 작업 노트일 뿐, 기준 그 자체가 아니다.
3. 화면에서 확인된 결과가 문서와 충돌하면, **문서를 고친다.**
4. 기존 문서와 기존 구현은 실패/학습 기록으로 취급한다.

즉, 이 문서의 역할은 “이상적인 설명”이 아니라, **어떻게 하면 원하는 화면을 실제로 만들 수 있는가**를 정리하는 데 있다.

---

## 기준 원칙

### 1. Source of Truth

이 작업의 최종 기준은 아래 순서로 둔다.

1. 실제 렌더링 결과
2. 사용자 피드백
3. 코드 구조
4. 문서

문서는 언제든 수정될 수 있다.

### 2. 구현 우선 원칙

- 먼저 작동하는 화면을 만든다.
- 그 다음 수치를 다듬는다.
- 그 다음 문서에 반영한다.
- 문서 일관성보다 **화면 설득력**이 우선이다.

### 3. 단계별 검증 원칙

이번 작업은 한 번에 완성하지 않는다.
각 기능을 분리해서 확인한 뒤, 점차 결합한다.

하지만 중요한 점은:

- phase는 **구현을 되돌아가기 위한 장치**가 아니라
- 원하는 결과로 가기 위한 **검증용 발판**이어야 한다.

즉, phase의 목적은 문서 분류가 아니라 **문제 원인 분리와 시각 검증**이다.

---

## 현재 작업 목표

메인 페이지 `섹션 1(Intro)` 의 카드형 이미지가 스크롤에 반응하여,
`섹션 2(About)`의 이미지 자리로 이동/변형되는 인터랙션을 구현한다.

다만 레퍼런스를 그대로 복제하지 않는다.
현재 프로젝트에 맞는 범위는 다음이다.

- 시작: `섹션 1`
- 종료: `섹션 2`
- 실제 moving card: 1개
- 실제 이미지 상태: 우선 2개까지 허용
- 모바일: 비활성 또는 정적 처리

---

## 현재까지 확인된 사실

### 1. 문서 해석보다 화면이 더 중요하다

이 작업에서는 여러 차례 문서가 먼저 정리됐지만,
실제 화면에서 보면 의도와 다르게 보이는 경우가 반복되었다.

따라서 앞으로는:

- “문서상 맞다”보다
- “화면상 그렇게 보이는가”를 기준으로 판단한다.

### 2. 축 용어는 쉽게 다시 꼬인다

`X축`, `Y축` 같은 이름은 계속 혼선을 만들었다.
그래서 구현 논의에서는 다음처럼 말한다.

- `좌우 회전`
- `위아래 회전`
- `가장 얇아지는 구간`
- `안착`
- `교체`

필요할 때만 코드 레벨에서 `rotateX/rotateY`로 번역한다.

### 3. “회전”이라는 말의 의미는 사용자 정의가 우선이다

현재 사용자 의도에서 `회전`은 단순 틸트가 아니라,
적어도 **한 바퀴 도는 체감**을 포함한다.

즉:

- 얕은 tilt만으로는 부족할 수 있다
- 단, full spin은 시각 왜곡을 크게 만들 수 있다
- 따라서 이 작업은 “정답 이론”보다 “어떤 구현이 원하는 인상으로 읽히는가”가 중요하다

### 4. 현재 문제는 라이브러리보다 연출 조합의 문제다

지금까지 드러난 주요 문제는 다음이었다.

- 회전 각도가 깊을 때 카드가 찌그러져 보임
- 180도/360도 구간에서 앞뒤 면 처리 문제가 생김
- 이동과 회전이 동시에 들어가면 카드가 사선으로 길게 늘어나 보임
- 카드가 섹션 2를 지나 다음 섹션까지 떠다님
- 단계 검증을 위해 남긴 placeholder가 실제 결과 판독을 방해하기도 함

이 문제들은 `framer-motion` 자체보다는,
다음 설계 요소와 더 관련이 크다.

- 이동 wrapper / 회전 wrapper 분리 여부
- 회전 깊이
- 회전 구간과 이동 구간의 관계
- 앞면/뒷면 구조
- 어떤 phase에서 무엇을 검증할지

---

## 현재 코드 상태 스냅샷

이 문서 작성 시점의 구현은 대략 이런 상태다.

- `PortraitMorphProvider` 존재
- `Intro / About` anchor 연결됨
- floating card 1개 존재
- `rotateY` 회전 실험 중
- `x / y` 이동 실험 중
- 앞면/뒷면 2면 구조 실험 중
- 이미지 교체는 아직 완성 단계 아님
- placeholder 숨김은 최종 단계로 미룸

즉, 현재는 **실험 중인 중간 상태**다.
이 상태를 이상적 기준으로 문서화하지 않는다.

## 작업 방식

### 큰 흐름

이 작업은 아래 순서로 진행한다.

1. 한 기능만 켜서 본다
2. 화면에서 인상을 확인한다
3. 문제가 있으면 수치가 아니라 구조 원인을 먼저 본다
4. 구조가 맞으면 다음 기능을 얹는다
5. 화면이 목표에 가까워지면 그때 문서를 갱신한다

### 가장 중요한 실무 규칙

- `회전`이 이상하면 각도만 만지기 전에 wrapper 구조를 본다
- `이동`이 이상하면 경로와 종료 범위를 본다
- `교체`가 이상하면 교체 타이밍보다 회전 피크를 먼저 본다
- `사라짐`이 생기면 backface / face 구조를 먼저 본다

---

## 권장 검증 순서

아래 순서는 **문서를 위한 phase**가 아니라,
실제 구현을 안정적으로 쌓아가기 위한 검증 순서다.

### Step 0 — 인프라 확인

목표:

- provider 연결
- main anchor 연결
- intro / about anchor 연결
- floating card가 실제로 렌더링되는지 확인

확인:

- 카드가 섹션 1 기준점에서 시작하는가
- 화면이 깨지지 않는가
- 모바일에서 비활성화되는가

### Step 1 — 회전만 검증

목표:

- 카드가 **화면에서 보이는 상태를 유지한 채**
- 스크롤에 따라 회전만 변하는지 확인

확인:

- 회전 인상이 의도대로 읽히는가
- 너무 틸트처럼 보이지 않는가
- 너무 찌그러지지 않는가
- 180도/360도에서 면 처리 문제가 없는가

주의:

- 이 단계에서는 “섹션 2까지 이동하는지”를 보지 않는다
- 회전이 보이지 않으면 이동을 얹지 않는다

### Step 2 — 이동만 검증

목표:

- 카드가 섹션 1 → 섹션 2로 이동하는 경로만 확인
- 회전은 잠시 약화하거나 꺼도 된다

확인:

- 카드가 어디까지 가야 하는지 명확한가
- 섹션 2를 지나 다음 섹션까지 떠다니지 않는가
- 안착 위치가 정확한가

주의:

- 이 단계에서는 회전 완성도보다 `종료 지점`과 `범위 제한`이 우선이다

### Step 3 — 회전 + 이동 결합

목표:

- 회전과 이동을 함께 걸었을 때도 결과가 자연스러운지 확인

확인:

- 카드가 회전하면서도 사선으로 길게 늘어나지 않는가
- 카드가 이동하며 너무 많이 찌그러지지 않는가
- 이동과 회전이 서로 싸우지 않는가

이 단계에서 문제가 생기면 먼저 확인할 것:

1. 이동 wrapper / 회전 wrapper가 분리돼 있는가
2. 회전 각도가 너무 깊지 않은가
3. 이동이 너무 초반부터 강하게 들어가지 않는가

### Step 4 — 안착 정리

목표:

- 섹션 2에서 카드가 어떤 자세로 끝나는지 고정

확인:

- 너무 평평한가
- 너무 기울었는가
- 최종 인상이 섹션 2와 어울리는가

### Step 5 — 이미지 교체

목표:

- 가장 얇아지는 순간 또는 회전 인상이 가장 강한 구간에서 이미지 교체

확인:

- 교체가 갑작스럽지 않은가
- 교체가 너무 늦지 않은가
- 교체 때문에 회전 인상이 깨지지 않는가

### Step 6 — placeholder/정적 카드 처리

목표:

- 최종 사용자에게는 카드가 2장, 3장 보이지 않게 정리

확인:

- floating card만 보이는가
- 검증용 흔적이 운영 화면에 남지 않는가

### Step 7 — 폴리시

목표:

- rotateZ
- border radius
- easing
- 미세 timing

## 구현 관점 핵심 제안

### 1. 이동과 회전은 wrapper를 분리하는 쪽이 낫다

현재까지의 문제를 보면,
`한 레이어에 x/y/rotateY를 동시에 넣는 구조`는 왜곡을 키울 가능성이 높다.

권장 구조:

- outer wrapper: 이동만
- inner wrapper: 회전만
- face layer: 앞면/뒷면 또는 이미지 상태

이렇게 해야 디버깅이 쉽고,
문제가 생겼을 때 이동/회전 책임을 분리할 수 있다.

### 2. full spin은 “원리상 가능”과 “예쁘게 보임”이 다르다

- 180° / 360°는 기술적으로 가능하다
- 하지만 카드형 이미지에서는 `edge-on collapse`와 `backface 문제`가 강하게 드러난다

즉, full spin은 단순한 값 튜닝 문제가 아니라,
의도적으로 면 구조/타이밍/원근을 설계해야 한다.

### 3. 구현은 실험해도 되지만, 실험과 목표를 혼동하지 않는다

지금은 시간이 충분하므로 full spin, two-face, deep rotation을 실험해도 된다.

하지만 항상 구분한다.

- 실험: 가능성 확인
- 목표: 실제로 설득력 있는 화면

실험 결과가 어색하면 과감히 버린다.

---

## 문서 갱신 규칙

앞으로 이 문서는 다음 경우에만 갱신한다.

1. 화면에서 확인된 결과가 바뀌었을 때
2. 구현 구조(예: wrapper 분리, two-face 채택)가 바뀌었을 때
3. 검증 순서가 바뀌었을 때

반대로 아래 경우엔 문서를 먼저 바꾸지 않는다.

- 아직 실험 중인 아이디어
- 화면에서 확인되지 않은 추정
- 구현되지 않은 이상적인 설명

---

## 현재 결론

이 작업의 핵심은 문서 정합성이 아니라,
**원하는 인상으로 읽히는 화면을 실제로 만드는 것**이다.

따라서 앞으로의 판단 기준은 다음 한 문장으로 고정한다.

> 문서는 구현을 돕기 위해 존재하며, 구현 결과가 문서보다 항상 우선한다.
