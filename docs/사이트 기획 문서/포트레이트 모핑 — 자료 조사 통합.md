## 2. 우리 결정 vs 실전 사례 — 검증 매트릭스

SSOT 문서의 핵심 결정을 외부 구현 사례에 비춰 점검한 결과.

| SSOT 결정 | 외부 근거 | 결론 |
|---|---|---|
| `rotateY` 좌우 회전 (rotateX 아님) | DAY_018, Codrops shape morph, Olivier Larose 모두 좌우(rotateY) | ✅ 표준 |
| 이동 wrapper / 회전 wrapper 분리 | Motion 공식 docs, Codrops 권장 구조 | ✅ 표준 |
| 동시 점진적 이동 (분리 분리 안 함) | Olivier Larose `useScroll`+`useTransform` 패턴 | ✅ 표준 |
| Full spin 회피, 피크 -34° 보수 시작 | DAY_018은 -180°까지 가지만 **두 면 구조**로 처리. 한 면 구조에선 -45~-60°가 한계 | ✅ 한 면 구조면 안전한 값 |
| `framer-motion` 유지 | layoutId는 우리 케이스 부적합(아래 4번), useScroll/useTransform은 정통파 | ✅ 도구 변경 불필요 |
| 모바일 비활성 | 대다수 사례 동일 결정 (세로 스택 구조에선 인상 약화) | ✅ 표준 |
| 이미지 교체를 가장 얇은 구간에서 | DAY_018의 90° 지점 swap, Codrops 동일 원리 | ✅ 표준 |
| 정적 anchor는 morph 활성 중 숨김 | 대다수 사례가 동일 (사용자에겐 floating 1장만) | ✅ 표준 |

→ 5번의 결정 변경 끝에 도달한 SSOT가 **외부 패턴과 거의 일치**. 자기 의심 그만하고 구현으로.

## 3. 표준 코드 패턴 (실전 사례에서 추출)

### 3.1 framer-motion `useScroll` + `useTransform` (정통파)

가장 우리 케이스에 가까운 패턴. Olivier Larose 스타일.

핵심 구조 (개념):

- **`useScroll({ target, offset })`** — 진입~탈출 구간을 0~1 progress로 매핑
- **`useTransform(progress, breakpoints, values)`** — 비단조 곡선 보간 (회전 4구간, swap 2구간)
- **wrapper 분리** — 바깥은 `{ x, y }` 이동, 안쪽은 `{ rotateY, rotateZ }` 회전, 가장 안쪽에 두 `<img>` cross-fade
- **이동은 0~1 전체 점진**, 회전은 비단조 (`[0, -18, -34, -20, -10]`), swap은 좁은 구간 (`[0.56, 0.70]`)

> 실제 적용 코드 골격은 **구현 로직 문서 §14** 참조. 본 문서는 *왜 이 구조인가*를 검증하는 자리이고, *어떻게 쓰는가*는 구현 로직이 SSOT.

### 3.2 회전 진행도 매핑 공식 (DAY_018, GSAP 버전)

```js
// startOffset: 카드별 시작 진행도 (stagger 효과)
const animationProgress = (progress - startOffset) / (1 / 3)
const frontRotation = -180 * animationProgress
const backRotation  = 180 - 180 * animationProgress
```

→ `progress`(0~1)를 `[startOffset, startOffset + 1/3]` 구간으로 정규화 후 보간. 우리는 stagger 안 쓰므로 `startOffset = 0`, 보간 구간 0~1 그대로.

### 3.3 두 면 구조 (full spin 또는 -90° 이상 회전 시)

```tsx
<motion.div style={{ rotateY, transformStyle: "preserve-3d" }}>
  <motion.div style={{ position: "absolute", backfaceVisibility: "hidden" }}>
    <img src={frontImg} />
  </motion.div>
  <motion.div style={{
    position: "absolute",
    backfaceVisibility: "hidden",
    transform: "rotateY(180deg)",
  }}>
    <img src={backImg} />
  </motion.div>
</motion.div>
```

회전 90° 지점에서 두 면이 모두 edge-on이 되며 자연스럽게 뒤집힘. **우리 SSOT는 -34° 피크라 두 면 구조 불필요**. 다만 회전 깊이를 -55° 이상으로 키우게 되면 한 면 구조의 swap이 들킬 위험이 있어 그때 두 면 구조로 갈아타는 게 안전.

## 4. 폐기된 가설 — 재시도 금지

### 4.1 ❌ Motion `layoutId`로 morph

**실패 이유**: `layoutId`는 mount/unmount 또는 state change 시 트리거. **continuous scroll progress로는 구동 안 됨.**

> "Layout animations are triggered when a component re-renders and its layout has changed."
> — Motion docs

→ 스크롤 50%일 때 카드가 50% 위치에 같은 비례 매핑은 layoutId로 못 만든다. 이 길은 막혀 있다.

### 4.2 ❌ react-flip-toolkit / react-morph

이산적 상태 전환(A → B) 라이브러리. 우리는 continuous scroll progress 기반이라 부적합.

### 4.3 ❌ "framer-motion이 문제"

3차 시도의 가설. 외부 사례를 보면 framer-motion으로 동일 효과를 깨끗이 구현한 사례가 다수. **문제는 항상 설계(wrapper 분리, 회전 깊이, 안착 타이밍)였다.** 라이브러리 교체로 해결되는 종류가 아니었음을 외부 사례로 재확인.

### 4.4 ❌ "X축 회전" 표기 → `rotateX`

2차 시도의 함정. 사용자 멘탈 모델("X축으로 빙글")과 표준 CSS(`rotateX = 위/아래 숙임`)가 어긋남. SSOT의 표준 축 매핑 표를 항상 참조.

### 4.5 ❌ Full spin (360°) 한 면 구조

4차 시도의 함정. mid-rotation에서 perspective 압축 + backface 노출. 외부 사례는 full spin을 두 면 구조로 처리하거나, 아예 -50° 미만의 비단조 틸트로 가는 두 가지 길만 있음.

## 5. 함정 인벤토리 (4차 시도 + 외부 사례 종합)

| # | 함정 | 증상 | 처방 |
|---|---|---|---|
| 1 | 한 wrapper에 x/y/rotateY 동시 | 사선으로 길게 늘어짐 | 이동 wrapper / 회전 wrapper 분리 |
| 2 | full spin 한 면 구조 | mid-rotation에서 찌그러짐, 거꾸로 이미지 노출 | 두 면 구조 OR 피크 -45° 미만 |
| 3 | swap 타이밍 너무 빠름 | swap이 회전 시작 전에 들킴 | swap을 회전 피크에 맞춤 (0.56~0.70) |
| 4 | swap 타이밍 너무 늦음 | 어색한 정지 후 갑작스런 변환 | 위와 동일 |
| 5 | 회전 부족해서 swap이 들킴 | 새 이미지 등장이 보임 | swap을 빠르게 X. **회전 피크를 키운다** |
| 6 | placeholder 운영 노출 | 사용자가 카드 2~3장 동시 보임 | morph 활성 중 anchor opacity 0 |
| 7 | 카드가 섹션 2 지나 떠다님 | 종료 지점 미정의 | offset 종료를 명시적으로 정의 |
| 8 | 이동이 회전 끝난 뒤 시작 | 정지 후 미끄러지는 어색한 인상 | 동시 점진적 이동 |
| 9 | reduced-motion 미처리 | 멀미 사용자 부담 | `prefers-reduced-motion: reduce` 시 정적 렌더 |
| 10 | 모바일에서 활성 | 세로 스택에서 인상 약화 | `min-width: 768px` 미만에서 비활성 |

## 6. 재시도 권장 시퀀스 (SSOT Phase 0~7 보강)

SSOT의 Phase 순서는 그대로 유지. 추가로 챙길 것만 정리.

### Phase 0 (인프라) — 추가
- **`prefers-reduced-motion` 처리부터 깔기**. 후반에 추가하면 reflow 디버깅 어려워짐.
- floating card는 일단 Intro 자리에 fixed positioning으로 고정. 모션 0.

### Phase 1 (회전만) — 추가
- 위치 변수를 명시적으로 0으로 락. `x = 0, y = 0`.
- 회전 인상이 안 보이면 각도를 키우기 전에 **wrapper 구조 의심**부터.

### Phase 3 (결합) — 강제
- wrapper 두 개 구조 강제:
  ```tsx
  <motion.div style={{ x, y }}>             // 이동
    <motion.div style={{ rotateY, rotateZ }}> // 회전
      <img />
    </motion.div>
  </motion.div>
  ```

### Phase 5 (이미지 교체) — 의사결정 분기
- swap이 들킨다 → **회전 피크 키우기** (-34° → -45° → -55°). swap 타이밍 더 만지지 말 것.
- -55°에서도 들킨다 → 두 면 구조로 갈아타기.

### Phase 6 (placeholder 처리) — 검증
- 데스크톱: floating 1장만 보임
- 모바일: 정적 anchor 정상 노출 (모션 비활성)
- 리사이즈: 두 모드 전환 부드러움

## 7. 도구 갈아타기 의사결정

언제 framer-motion → GSAP 전환을 고려할지.

**전환 안 함** (현재 권장):
- SSOT 결정대로 진행, Phase 0~7 1회 통과
- swap 타이밍·회전 피크만 Phase 5에서 미세 조정

**전환 검토 트리거**:
- Phase 3 결합 단계에서 동일 wrapper 구조로 3회 이상 의도와 다른 결과
- framer-motion의 `useScroll` `offset` 계산이 sticky/fixed 컨테이너와 충돌해 디버깅 1일+ 소요
- 둘 다 GSAP+ScrollTrigger로는 알려진 해결책이 있는 케이스

**전환 비용**:
- GSAP+ScrollTrigger 학습/이식: 0.5~1일
- 기존 framer-motion 의존성 유지(다른 인터랙션이 쓰는 중) — 둘 다 번들에 들어감 (~50KB 추가)
- 그 외 컴포넌트는 framer-motion 그대로

→ **현 단계에선 전환 비용 정당화 어려움.** Phase 0~7 한 번 깨끗이 돌리고 결정.

## 9. 외부 사례에서 얻은 추가 디테일 (SSOT 반영 검토)

SSOT에 없지만 사례에서 발견한 것들. 채택 여부는 구현 후 판단.

### 9.1 `scrub: 1` 패턴 (GSAP 용어)
스크롤과 애니메이션을 1:1로 직결. framer-motion에선 `useScroll`이 기본적으로 같은 동작. 별도 처리 불필요.

### 9.2 Lenis와의 동기화
DAY_018은 Lenis 사용. 우리도 이미 Lenis 사용 중. framer-motion `useScroll`은 native scroll과 Lenis 모두 호환. **변경 사항 없음**.

### 9.3 카드 stagger
DAY_018은 카드 여러 장이라 stagger 적용. **우리는 단일 floating 카드라 불필요**.

### 9.4 perspective 컨테이너
3D 회전 시 부모에 `perspective: 1000px` 권장. 두 면 구조로 갈아탈 경우 명시적으로 설정.

```tsx
<div style={{ perspective: "1200px" }}>
  <motion.div style={{ rotateY, transformStyle: "preserve-3d" }}>
    ...
  </motion.div>
</div>
```

한 면 구조에선 perspective 없어도 작동(rotateY가 화면 평면에 그대로 그려짐). 깊이감 더 주려면 추가.

### 9.5 will-change / transform-gpu
`transform-gpu` 클래스 또는 `will-change: transform` 추가 시 GPU 합성으로 60fps 유지. 모바일 비활성 케이스라 데스크톱만 신경 쓰면 됨.

## 10. 최종 메시지

이 문서는 **SSOT를 보강하는 외부 검증 자료**다. SSOT의 결정을 흔들기 위함이 아니다.

- SSOT가 결정한 것: 5차 재구현의 **What**
- 본 문서가 보강한 것: 그 결정의 **외부 검증**과 **실전 코드 패턴**

함정 인벤토리(5번)와 권장 시퀀스(6번)만 옆에 두고 SSOT의 Phase 0~7을 한 번 깨끗이 돌리는 것 — 그게 5차 재시도의 작업 흐름이다.
